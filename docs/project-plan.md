## 1. 计划说明

本文档把 [需求规格说明书](./requirements.md)、[概要设计说明书](./design.md)、[详细设计说明书](./detail-design.md)、[测试计划](./test.md) 和 [部署说明](./deploy.md) 转换为可执行的项目路线、任务编号、验收门槛与需求追踪关系。

参考实现为 [change-42-yhmm/quota-float](https://github.com/change-42-yhmm/quota-float)。参考项目只用于理解 Tauri 2 桌面工程、浮动窗口、托盘、单实例、偏好持久化和测试组织方式，不作为 QuotaGlance 的代码或验收基线。

计划中的“完成”具有严格含义：代码已合入、自动化检查通过、相关实机或安全验证完成，并留下可追溯证据。仅创建文件、写出接口、在参考项目中通过测试或在单一开发机上运行，均不能把对应里程碑标记为完成。

状态统一使用以下取值：

| 状态 | 含义 |
|---|---|
| 规划 | 已纳入范围，尚未开始 |
| 待实施 | 已排入当前迭代，尚无可验收结果 |
| 进行中 | 已开始实现，尚未满足全部验收条件 |
| 阻塞 | 因外部条件无法继续，已记录证据、影响和解除条件 |
| 待验收 | 实现和自测完成，等待独立复核 |
| 已完成 | 验收通过，证据已归档 |
| 延后 | 经批准移出当前版本，并记录影响 |

### 2.1 本轮实施边界

本轮已实施 **M0 可行性验证程序骨架 + M1 可构建工程骨架**，并提前启动了部分 M2/M3 纵向切片，具体含义是：

- 已建立可锁定安装依赖、可执行前端和 Rust 检查、可构建 Tauri 调试桌面壳的工程；
- 已建立 App Server 进程与 `stdio` JSONL 协议的模块边界、最小握手、只读允许列表和合成输入测试；
- 已建立 IPC、窗口/托盘骨架和 React 额度 UI，浏览器演示 fixture 与 Tauri 真实读取路径分离；
- 当前 Windows 登录账号已通过 Codex 桌面受管运行时完成只读 POC；M0 仍因固定版本、完整认证矩阵、macOS 和合法 bundled sidecar 分发证据而部分阻塞，不读取 `auth.json` 或 Token；
- 已为后续缓存、调度和持久化提供接口，但不在骨架中伪造“真实额度已可用”；
- 所有未完成的其他认证模式、双平台、签名和发布事项继续保持“规划”“待验证”或“阻塞”。

本轮不把 M2 的完整额度服务、M3 的正式浮球体验、M4 的双平台适配、M5 的发布工程或 M6 的正式发布提前包装成已交付能力。

## 3. 项目目标与非目标

### 3.1 1.0.0 目标

1. 通过受控 Codex App Server 的公开本地协议，只读获取账号模式和 ChatGPT 额度。
2. 动态展示额度桶、额度窗口、剩余比例、重置时间、Credits、重置次数和数据新鲜度。
3. 在数据源异常、断网、登录失效和协议变化时安全降级，保留最后成功快照，不制造 `0%`。
4. 提供可恢复的浮球、展开卡片、隐藏状态、托盘、置顶、鼠标穿透、位置记忆和单实例体验。
5. 支持简体中文、英文，以及跟随系统、极光、石墨、纸白、日落珊瑚、蜂蜜琥珀和玫瑰铜夜主题。
6. 在 Windows 11 x64、macOS 13+ Intel 和 Apple Silicon 上完成同一功能口径的实机验收。
7. 建立可重复的测试、构建、签名、公证、更新和回滚流程。
8. 保持本地优先、无默认遥测、无额度历史、无账号写操作的隐私与安全边界。

### 3.2 1.0.0 非目标

- 不自建登录、云端账号、同步服务或团队管理后台；
- 不读取 `auth.json`，不访问系统 keyring，不持有、解析或记录 Token；
- 不调用登录、登出、购买、兑换重置次数等账号写方法；
- 不使用私有 Wham HTTP 接口作为默认或静默降级路径；
- 不保存额度历史，不提供趋势、预测耗尽时间或本地 token 推算；
- 不支持多账号自动切换、其他 AI Provider 或移动端；
- 不使用 MySQL、Redis、消息队列或本地 REST 服务；
- 不承诺 Windows ARM64，也不把 Windows 10 作为 1.0 阻塞性支持基线；
- 不在 M0/M1 骨架阶段追求最终视觉、完整托盘菜单或正式安装包。

## 4. 交付策略

项目采用“技术风险优先、纵向切片、证据驱动”的推进方式：

1. 先证明 App Server 主链路和合法分发路径，再扩大 UI 投入；
2. 协议、Provider、领域、应用服务和表现层保持单向依赖；
3. 真实账号只用于受控 POC，日常自动化使用假进程和脱敏 fixture；
4. 每个里程碑设置进入条件和退出条件，未过门禁不得把下游功能标为完成；
5. Windows 与 macOS 的平台行为分别验收，交叉编译不能替代实机验证；
6. 实现状态同步回需求、测试、部署和更新日志，避免计划与仓库事实分离。

## 11. 排期与迭代节奏

### 11.1 建议节奏

- 迭代长度：10 个工作日；
- 计划更新：迭代开始时确认范围、负责人和依赖；
- 技术同步：工作日按阻塞需要进行，避免为汇报制造无效会议；
- 中期检查：第 5 个工作日复核门禁、风险和估算；
- 演示与验收：迭代最后 1～2 天，只演示可复现结果；
- 回顾：记录偏差、返工原因、待改进项和下轮输入。

### 11.2 估算规则

- 人日包含实现、自测、审查和文档同步，不只计算编码时间；
- 外部等待单独记录，不把证书、设备或账号等待伪装成开发进度；
- 超过原估算 50% 的任务必须拆分或重新评估；
- 协议、安全和跨平台任务至少保留 20% 的验证缓冲；
- 里程碑估算在 M0 结论出来后重新基线化一次。

## 12. Definition of Done

### 12.1 单个任务完成标准

一个任务只有同时满足以下条件才可标记“已完成”：

- 验收条件明确，并与需求或技术债编号关联；
- 实现遵守模块依赖、类型、错误处理和中文说明规范；
- 新行为有对应单元、契约、集成或实机测试；
- 所有相关自动化检查通过，无被忽略的失败；
- 安全边界没有扩大，日志、fixture 和错误信息不含敏感数据；
- 代码经过至少一名非作者复核；
- 影响到接口、配置、存储、部署或用户行为时，相关文档已同步；
- 验收证据链接到任务记录。

### 12.2 迭代完成标准

- 迭代承诺任务均为“已完成”，或经批准转为下一迭代并记录原因；
- 当前平台的类型检查、lint、测试、Rust 检查和构建全部通过；
- 新增阻塞、风险、架构决策和需求变化已登记；
- 未完成事项没有被写入 README、changelog 或演示文案作为已支持能力；
- 迭代验收记录包含提交号、环境、命令、结果和负责人。

### 12.3 里程碑完成标准

- 里程碑退出条件逐项通过；
- 映射到该里程碑的需求达到约定状态，并具有证据；
- 阻断缺陷为零，高风险问题有明确关闭结论；
- 下游里程碑依赖的接口与行为已冻结或完成版本说明；
- 技术负责人、测试负责人和对应产品/平台验收人签字确认。

### 12.4 1.0.0 发布完成标准

- FR-001～FR-041、NFR-001～NFR-018 全部达到“已验收”；
- Windows 11 x64、macOS Intel、macOS Apple Silicon 实机矩阵通过；
- 正式安装包、sidecar 和更新产物的签名、哈希、公证及 stapling 可验证；
- 更新、篡改拒绝、失败恢复、覆盖升级和卸载通过；
- secret scan、依赖审计、SBOM、LICENSE/NOTICE 与隐私审查通过；
- 测试报告、部署说明、用户手册、故障排查、隐私、安全和 changelog 与发布版本一致；
- 正式产物不可变，发布后观察、回滚和支持责任人已明确。

## 14. 变更管理

### 14.1 变更触发条件

以下情况必须提交变更记录：

- P0 功能、验收指标、目标平台或安全边界发生变化；
- App Server 方法、Schema、sidecar 来源或许可结论变化；
- 新增依赖、Tauri Capability、外部网络访问或本地持久化数据；
- 里程碑日期预计偏差超过 20%，或任务估算增加超过 50%；
- 需要启用 Legacy Provider、私有接口或实验 API；
- 签名、更新、隐私或合规方案发生变化。

### 14.2 变更流程

1. 提交变更说明，包含背景、目标、非目标和替代方案；
2. 分析对需求、架构、接口、数据、安全、测试、平台、排期和发布的影响；
3. 产品负责人确认范围，技术负责人确认方案，安全负责人确认边界，测试负责人确认验收；
4. 对架构性决定新增或更新 ADR；
5. 先更新权威需求和设计文档，再调整任务、估算和里程碑；
6. 实施后补充测试证据和 changelog；
7. 被拒绝或延后的变更也要记录理由，避免重复讨论。

未经上述评审，不得：

- 为赶进度直接读取 `auth.json`、keyring 或 Token；
- 从 WindowsApps 复制或提权执行内部二进制；
- 把私有 Wham 接口设为默认数据源；
- 放宽 WebView Shell、文件或网络权限；
- 删除失败测试、降低安全门槛或把未知值显示为 `0%`；
- 将 P1/P2 功能插入当前迭代并挤占 P0 门禁。
