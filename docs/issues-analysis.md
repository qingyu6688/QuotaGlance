# QuotaGlance 未完成问题分析与解决方案

> 分析日期：2026-07-13  
> 适用版本：`0.1.x` 开发与 prerelease 基线
> 文档维护：maorongkang@gmail.com

## 1. 分析结论

未完成事项不能统一归类为“代码没写完”。当前问题分为三类：

1. **本地可闭环问题**：不依赖真实账号、证书或外部发布授权，可以通过实现和自动化测试解决。
2. **需要仓库或独立环境的问题**：代码可以准备，但必须在 GitHub、Windows 11、macOS 或干净安装环境留下证据。
3. **外部阻塞问题**：需要合法 sidecar 来源、再分发许可、专用账号或签名资产，当前不能通过技术绕过。

本轮优先解决会影响数据正确性、请求风暴、故障恢复和用户设置丢失的问题。当前 Windows 账号链路已通过 Codex 自身 App Server 验证；合法 bundled sidecar、固定版本和发布许可继续保持阻塞，不能用 WindowsApps、凭据文件或私有接口绕过。

## 3. 本轮解决方案设计

### 3.1 刷新与恢复

- 手动刷新、账号通知和额度通知始终绕过缓存，确保用户动作和失效信号能读取最新数据。
- 可见/隐藏安全重同步在最后成功读取后 30 秒内命中内存缓存。
- 存在 `nextRetry` 时，重试优先级高于最近成功缓存，避免“刚成功过一次”掩盖随后发生的连接故障。
- 同类读取继续由应用级刷新锁合并，底层同时最多一个账号/额度读取序列。
- 可重试失败按 `30 秒 → 1 分钟 → 2 分钟 → 5 分钟 → 15 分钟 → 30 分钟` 退避；成功后清零。
- 不可重试错误暂停自动刷新，保留手动重试入口。

### 3.2 偏好持久化

```text
读取 preferences.json
→ 检查 256 KiB 上限与 schemaVersion
→ 反序列化并执行领域校验
→ 当前文件损坏时尝试 preferences.json.bak
→ 两者均无效时使用安全默认值并返回恢复原因
```

保存流程：

```text
校验完整 Preferences
→ 写入 preferences.json.tmp
→ flush + sync_all
→ 当前文件有效时更新 .bak
→ 替换 preferences.json
→ 更新内存状态并发布事件
```

高于当前应用的 Schema 会进入只读保护，旧版本不会覆盖未来版本文件。

### 3.3 依赖审计

2026-07-13 执行 `cargo audit`：

- 扫描 468 个 Cargo.lock 依赖；
- 未发现 RustSec 漏洞；
- 发现 17 个允许警告，其中 Linux GTK3 绑定为 Tauri 跨平台目标的间接依赖，`glib 0.18.5` 有一项 RustSec 不健全警告，其余主要是停止维护警告；
- QuotaGlance 1.0 当前目标为 Windows/macOS，但仍需在依赖升级时复查，不能把“无漏洞”解释为“无供应链风险”。

## 4. 验收边界

本轮可以关闭的是本地代码与测试缺口，不能关闭以下门禁：

- M0 真实账号和 sidecar 合法来源；
- GitHub 远程 CI 与独立复核；
- Windows 11/macOS 实机；
- 安装器、签名、公证和安全更新；
- 真实协议版本与字段兼容矩阵。

这些项目只有在外部条件到位并留下可追溯证据后才能从“阻塞/待实施”改为“已完成”。
